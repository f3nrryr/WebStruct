ADR - architecture decision remarks.  
  
Для API расчётных моделей предлагается следующая архитектура...  
  
С точки зрения масштабирования при необходимости можно будет просто добавлять новые инстансы API через k8s либо через другой фасад-балансировщик.  
  
Но это может быть нужно, если будут активно пользоваться.  
  
По авторизации - для входа/выхода ASP Identity.  
  
Для безопасности - Bearer? Хотя если внутренняя сеть, то зачем?..  
  
Кэшировать как будто бесполезно. Хотя если нагрузка, то IMemoryCache или Redis'ка в помощь.  
  
UsersRoles идут вместе, т.к. одно без другого бесполезно. Т.к. на кафедре вряд ли есть куча мощностей,  
делаем и UsersRoles, и CompModels в рамках одной API.  
  
Деление по слоям обосновываю так: DAL - изменяемый. Зеркало таблиц и сущностей б/д. Можно перегенерить в случае изменений в б/д.  
  
Services - +1 к Repo со стороны б/д. Логические условия, а на выход, по сути, модели, готовые к отрисовке.  
  
В отдельные проекты вынесены WebStruct.?.Contracts для удобства - это контракты API наружние.  
  
Repositories - точка входа к работе с б/д. Вся работа с б/д через Repository. Единая точка входа, переиспользуемость. Возможность гибкой замены б/д  
и тестирования.  
  
AutoMapper не цепляю осознанно, т.к. его может быть неудобно дебажить.  
  
Docker цеплять без k8s смысл вряд ли будет.  
  
На выход у моделей - JSON-string с целью "тупого фронтенда", ибо зачем его перегружать думательными операциями, если с б/д и расчетами непосредственно он
не работает.  
  
ORM вместо RAW-SQL для удобства отладки проблем.  
  
traceId = отслеживание запроса в логах.
